# API 设置界面 PRD

## 1. 背景

当前项目的 API 设置能力主要集中在 [src/components/SettingsDialog.tsx](F:/AI%20HOME/CODEX/%E6%89%B9%E9%87%8F%E5%B7%A5%E4%BD%9C%E5%8F%B0-%E6%9E%B6%E6%9E%84%E6%94%B9%E9%80%A0%E5%A4%87%E4%BB%BD-20260423-165902/src/components/SettingsDialog.tsx) 中，已经支持：

- 平台切换
- 文本 API / 生图 API 配置
- 模型选择
- 连接测试
- 额度查询
- 远程模型拉取
- 配置导入导出
- 下载目录设置
- 并发设置
- 生成日志入口

但当前界面把“API 配置”和“系统杂项设置”放在同一个弹窗里，信息密度高，结构偏长，对不懂技术的用户不够友好。

本 PRD 目标是定义一个更清晰、更稳、更容易使用的 API 设置界面，用于后续 UI 重构或功能补齐。

## 2. 目标

- 让用户能在 1 分钟内完成一个平台的 API 配置。
- 让用户能清楚区分“当前在配置哪个平台”。
- 让用户能快速完成“填地址、填 Key、选模型、测试连接、保存”这条主流程。
- 保留当前项目已有的导入、导出、额度查询、模型拉取等能力，但收纳得更清楚。
- 降低误配风险，尤其是文本接口和生图接口混淆的问题。

## 3. 非目标

- 本次不改底层图片生成路由逻辑。
- 本次不改 `batchRunner.ts` 的平台调用规则。
- 本次不新增新的平台类型。
- 本次不改项目导入导出协议本身，只改界面组织方式和交互要求。

## 4. 当前问题

### 4.1 信息堆叠过长

当前弹窗是单列长表单，平台、接口、测试、下载目录、并发都堆在一起。用户需要上下滚动，容易丢失上下文。

### 4.2 主流程不够突出

用户最常做的是：

1. 选择平台
2. 填 API 地址
3. 填 API Key
4. 选模型
5. 测试连接
6. 保存

但当前这个主流程被导入、导出、日志、下载目录、并发等功能打散了。

### 4.3 文本和生图的关系不够直观

当前支持：

- 文本和生图共用一个地址 / Key
- 也支持生图单独地址 / 单独 Key / 单独路径

这本身是对的，但界面没有把“默认继承”和“单独覆盖”的关系讲清楚。

### 4.4 平台切换后的心理模型不够稳定

当前每个平台会分别保存配置，这是对的。但用户切换平台后，很难一眼确认“我现在修改的是这个平台自己的配置，而不是全局配置”。

### 4.5 API 设置与系统设置混在一起

下载目录、任务并发、生成日志入口都放在同一个设置弹窗里，用户会把“平台接入配置”和“软件行为配置”混成一类。

## 5. 设计原则

- 主任务优先：优先服务“配置平台并跑通”。
- 对不懂技术的用户友好：少术语，多提示，多默认值。
- 当前平台强感知：任何时刻都要明确“正在编辑哪个平台”。
- 默认安全：尽量减少用户因为留空、填错、误切平台导致的失败。
- 分组清晰：把 API、验证、配置管理、系统项拆开。

## 6. 页面定位

本页面是“平台接入设置页”，不是“大而全的系统控制台”。

建议在产品文案上统一使用：

- 页面标题：`API 设置`
- 副标题：`配置当前平台的文本与生图接口`

不建议继续使用过泛的 `系统设置` 作为这个界面的唯一标题。

## 7. 信息架构

建议将页面拆成 4 个模块，按优先级排序：

1. 平台选择区
2. API 配置主表单
3. 连接验证与辅助操作区
4. 配置管理与高级设置区

其中：

- `1 + 2 + 3` 属于主流程
- `4` 属于次要操作

## 8. 布局方案

### 8.1 桌面端布局

建议使用 `左右两栏 + 底部固定操作区`：

- 左栏宽度约 220 到 260
- 右栏为主配置区
- 底部固定保存区，不随滚动消失

左栏：

- 平台列表
- 当前平台状态
- 简短帮助文案

右栏：

- 当前平台表单
- 验证操作
- 配置管理
- 高级设置

底部固定区：

- `保存设置`
- `取消`
- 保存状态提示

### 8.2 移动端 / 小屏布局

建议改成单列，但顺序保持一致：

1. 平台选择
2. API 配置
3. 验证操作
4. 配置管理
5. 高级设置
6. 底部固定保存按钮

要求：

- 保存按钮始终可见
- 平台切换不应导致页面跳顶
- 模型下拉不要超出屏幕

## 9. UI 方案

### 9.1 整体风格

延续当前项目已有的浅米色、白卡片、圆角、轻阴影风格，不另起新主题。

要求：

- 背景延续当前 `#F9F8F6` 一类暖白底
- 卡片使用白色或半透明白色
- 圆角维持当前 20 到 24 的柔和风格
- 重要按钮继续使用当前主按钮色

### 9.2 左栏平台卡

每个平台显示为一张简短卡片，包含：

- 平台名
- 当前是否已配置
- 当前文本模型摘要
- 当前生图模型摘要

状态规则：

- 未填写 Key：显示 `未配置`
- 已填写但未测试：显示 `待验证`
- 测试成功：显示 `已连接`
- 测试失败：显示 `连接失败`

当前选中平台要有明显高亮。

### 9.3 右栏主表单卡片

建议拆成以下卡片：

#### A. 文本接口卡

- 文本 API 地址
- 文本 API Key
- 文本模型

#### B. 生图接口卡

- 生图 API 地址
- 生图接口路径
- 生图 API Key
- 生图模型

#### C. 连通性与能力卡

- 测试 API 连接
- 查询额度
- 拉取模型
- 状态反馈

#### D. 配置管理卡

- 导入当前配置文件
- 导出当前平台配置
- 导出全部平台配置

#### E. 高级设置卡

- 结果图下载目录
- 任务并发
- 查看生成日志

### 9.4 字段表现

要求：

- 输入框使用统一高度
- 每个字段有清晰标题
- 占位文案要写“示例值”和“留空后的默认行为”

例如：

- 生图 API 地址占位文案：`留空则跟随文本 API 地址`
- 生图 API Key 占位文案：`留空则跟随文本 API Key`

### 9.5 状态反馈

成功提示：

- 用绿色浅底状态条
- 文案简短直接

失败提示：

- 用红色浅底状态条
- 展示接口返回的主要错误
- 支持长文本换行，不截断

加载中：

- 按钮内显示 `测试中...`、`查询中...`、`拉取中...`
- 同时禁用重复点击

## 10. 功能需求

### 10.1 平台切换

用户可以在以下平台之间切换：

- `yunwu`
- `comfly-chat`
- `openai-compatible`
- `gemini-native`
- `custom`

切换规则：

- 每个平台有独立配置
- 切换前先把当前表单改动同步到本地临时状态
- 切换后展示目标平台自己的配置
- 未保存前，不写入最终项目状态

### 10.2 表单编辑

支持编辑：

- 文本 API 地址
- 文本 API Key
- 文本模型
- 生图 API 地址
- 生图 API 路径
- 生图 API Key
- 生图模型

规则：

- 文本地址改动时，可同步更新“总地址”概念
- 生图地址和 Key 可独立覆盖
- 模型输入支持手填，也支持从远程列表中选

### 10.3 模型选择

支持两种来源：

- 内置模型列表
- 远程拉取模型列表

要求：

- 支持搜索过滤
- 默认最多展示前 12 条匹配项
- 支持手动输入未出现在列表中的模型名

### 10.4 连接测试

用户点击 `测试 API 连接` 后，系统应：

- 按当前平台规则发起测试请求
- 优先测试当前文本模型
- 返回成功或失败结果

成功反馈：

- `连接正常`
- 如有额外能力信息，可显示 `已获取 N 个模型`

失败反馈：

- 直接显示主要错误原因
- 不要只提示“失败”，要告诉用户可能是地址、Key、网络还是模型问题

### 10.5 额度查询

仅对支持的平台显示：

- `yunwu`
- `comfly-chat`

显示规则：

- 不支持的平台不展示该按钮
- 结果展示在状态卡中

### 10.6 远程模型拉取

用户点击 `拉取模型` 后，系统应：

- 使用当前表单中的地址和 Key 请求模型列表
- 自动去重
- 按“文本模型 / 图片模型”拆分
- 更新当前页面可选项

### 10.7 配置导入导出

支持：

- 导入配置
- 导出当前平台配置
- 导出全部平台配置

要求：

- 明确提示导入的是“当前平台”还是“全部平台”
- 导入成功后立即刷新当前表单
- 导出文件名可读
- 继续沿用现有加密配置方案

### 10.8 下载目录设置

支持：

- 选择下载目录
- 重新选择目录
- 清除目录

要求：

- 如果环境不支持目录写入，要明确提示会回退为 ZIP 下载
- 当前目录名必须可见

### 10.9 并发设置

支持设置任务并发数。

要求：

- 仅允许 1 到 10
- 超出范围时给出即时提示
- 文案说明“过高可能触发平台限流”

### 10.10 日志入口

保留 `查看生成日志`，但定位为排障辅助入口，不应压过主流程。

## 11. 交互规则

### 11.1 保存前状态

用户在弹窗中编辑时，改动先保留在本地表单态，不直接写入全局。

### 11.2 保存时机

点击 `保存设置` 后，统一写入：

- 当前平台配置
- 当前选中平台
- 下载目录相关配置
- 并发设置

### 11.3 离开提醒

如果用户有未保存改动就关闭弹窗，建议弹出二次确认：

- `放弃修改`
- `继续编辑`

### 11.4 按钮优先级

主按钮：

- `保存设置`

次按钮：

- `测试 API 连接`
- `拉取模型`
- `查询额度`

弱按钮：

- `导入`
- `导出当前`
- `导出全部`
- `查看生成日志`

## 12. 文案要求

要求所有文案都尽量口语化、少术语。

建议示例：

- `文本 API`
- `生图 API`
- `留空则跟随文本 API`
- `先测试连接，再开始批量生成`
- `当前平台的地址、Key 和模型会单独保存`

不建议大量使用：

- `Endpoint`
- `Credential`
- `Payload`

## 13. 验收标准

### 13.1 布局验收

- 桌面端能明确看出左侧平台、右侧配置的两层结构
- 小屏下不出现严重拥挤或按钮跑出屏幕
- 保存按钮始终容易找到

### 13.2 UI 验收

- 当前平台高亮清晰
- 文本接口和生图接口视觉上明显分开
- 成功、失败、加载中三种状态明显不同
- 导入导出和主表单不会视觉抢占

### 13.3 功能验收

- 切换平台不会丢失当前未保存编辑态
- 保存后能正确恢复到项目状态
- 测试连接、查询额度、拉取模型可按平台正常工作
- 导入后表单内容立即更新
- 导出当前与导出全部语义清晰

## 14. 实施建议

建议按以下顺序实施：

1. 先把当前单列弹窗重构为“平台栏 + 主配置区”
2. 再拆分主配置区卡片，把 API 主流程前置
3. 再补未保存提醒、平台状态标记、字段说明
4. 最后再优化移动端和状态文案
