# 图片顶部清空与底部内嵌控件统一 PRD

## 1. 背景

上一轮已经把大量文字信息下沉到卡片下半部，但你这次指出了两个更严格的问题：

### 1.1 图片上方仍然留了东西

虽然编号、状态、标题等已经开始往下沉，但图片区顶部仍然保留了操作控件：

- 拖拽
- 复选框
- 查看按钮
- 删除按钮

这会导致图片区顶部仍然不是“纯图”。

你的要求已经非常明确：

- 图片上方区域不要留任何东西
- 所有图内控件只允许在图片内部靠下区域出现

### 1.2 复选框和上传按钮样式内外不统一

当前存在一个明显的系统问题：

- 卡片里的复选框样式一套
- 图片区里的复选框又是一套
- 参考图区里的上传按钮又是另一套

结果是：

- 同一张卡片内部控件语言不统一
- 展开态和收起态的细节风格仍然割裂

这次要解决的不是单个按钮，而是整个“控件系统一致性”。

## 2. 目标

- 图片顶部彻底清空，不再保留任何控件或信息
- 图内控件统一沉到图片内部靠下区域
- 编号和状态继续下沉到卡片下半部
- 统一复选框、上传按钮等基础控件样式
- 让展开态和收起态继续向同一套组件系统收拢

## 3. 非目标

- 本次不改批次生成逻辑
- 本次不改 Lightbox 核心逻辑
- 本次不改参数区、提示词区、参考图区的数据结构
- 本次不改页面整体布局

## 4. 核心原则

### 4.1 图片顶部绝对清空

图片区顶部不再允许出现：

- 按钮
- 文案
- 状态
- 编号
- 勾选
- 删除

顶部就是纯图片区域。

### 4.2 控件下沉

如果控件必须留在图片内部，只允许出现在图片内部靠下区域。

### 4.3 信息归下

编号、状态、标题、返回数量等信息全部归入卡片下半部。

### 4.4 基础控件统一

复选框、上传按钮、轻操作按钮必须属于同一控件体系。

## 5. 问题定义

## 5.1 现在图片区上方仍然不干净

当前图片区顶部仍然承担：

- 操作入口
- 风险按钮
- 辅助控制

这会让用户第一眼先看到 UI，而不是图。

## 5.2 图内控件分布还不够稳定

当前控件位置比较分散：

- 有的在上
- 有的在下
- 有的在图内
- 有的在图外

这会让卡片缺少统一秩序。

## 5.3 基础交互控件语言不统一

最典型的是：

- 复选框风格不统一
- 上传按钮风格不统一
- 内部和外部的按钮边框、圆角、透明度也不一致

这说明组件系统还没有收口。

## 6. 目标结构

新的卡片结构定义如下：

### 6.1 图片区

图片区只包含：

- 主图
- 图片内部靠下的必要控件层

### 6.2 图片区顶部

必须为空。

### 6.3 图片区底部

承担图内控件：

- 缩略图
- 尺寸标签
- 查看按钮
- 删除按钮
- 必要的拖拽/勾选入口（如果仍需留在图内）

### 6.4 下半部信息区

统一承载：

- 编号
- 状态
- 标题
- 返回数量
- 描述
- 参数
- 提示词
- 错误与操作区

## 7. 详细方案

## 7.1 图片顶部清空

这是本次第一优先级。

### 必须移出的内容

- 拖拽入口
- 复选框
- 查看按钮
- 删除按钮
- 编号
- 状态

### 结果

- 图片顶部不再有任何视觉元素
- 用户第一眼只看到图片主体

## 7.2 图内控件改到底部

若某些控件仍需要贴图存在，则统一进入图片内部底部区域。

### 可留在图内底部的控件

- 缩略图切换
- 尺寸标签
- 查看按钮
- 删除按钮
- 必要时保留拖拽/勾选

### 推荐布局

- 左下：拖拽 + 勾选 + 缩略图
- 右下：尺寸标签 + 查看按钮 + 删除按钮

### 原则

- 全部靠底
- 不再靠顶
- 不遮挡主图主体区域

## 7.3 编号和状态继续下沉

### 新规则

- 编号不再出现在图上
- 状态不再出现在图上
- 统一进入下半部信息头

### 信息头结构

- 第一行：编号 + 状态 + 返回数量
- 第二行：标题

### 目标

- 图上彻底无正文信息
- 下半部成为唯一阅读区

## 7.4 复选框样式统一

当前复选框的内外观感还不一致。

### 统一要求

- 圆角一致
- 尺寸一致
- 边框强度一致
- 选中态图形一致
- 背景和 hover 反馈一致

### 适用范围

- 卡片图片区的复选框
- 卡片其它位置若有复选交互

### 目标

- 用户看到的所有复选框都像同一个组件

## 7.5 上传按钮样式统一

当前上传按钮在参考图区的风格与其它轻操作按钮不完全同系统。

### 统一要求

- 与轻操作按钮同一圆角体系
- 与轻操作按钮同一边框强度
- hover 反馈一致
- 尺寸规则一致

### 适用范围

- 参考图区上传按钮
- 其它图片上传入口

### 目标

- 上传按钮不再像单独的一类 UI

## 7.6 图内按钮系统统一

这次顺带统一所有图内轻按钮。

### 统一对象

- 查看按钮
- 删除按钮
- 拖拽入口
- 复选框

### 统一维度

- 尺寸
- 圆角
- 透明背景强度
- hover 反馈
- 边缘间距

## 7.7 展开/收起继续统一

这次要继续推进两种状态的统一。

### 统一方向

- 两者都遵守“上图下文”
- 两者的图片顶部都清空
- 两者的信息头都在下半部
- 两者的基础控件都使用同一视觉系统

### 差异只保留在

- 展开态图片更大
- 展开态下半部信息更多

## 8. 建议一起处理的优化点

### 优化点 1

统一图内底部控件带的留白

底部缩略图、尺寸、按钮之间的边距应统一。

### 优化点 2

统一信息头与图片区距离

下半部信息头与图片区之间的间距，在展开和收起态都应一致。

### 优化点 3

返回数量并入信息头标准样式

不要再让返回数量有单独一套表现。

### 优化点 4

删除按钮危险态继续明确

虽然下沉到底部，但 hover 态仍要清楚区分高风险操作。

### 优化点 5

缩略图和底部按钮的视觉优先级再梳理

避免底部控件过多后彼此抢注意力。

## 9. 验收标准

### 必须满足

- 图片顶部不再保留任何东西
- 编号和状态全部下沉到卡片下半部
- 图内控件统一改到图片内部靠下区域
- 复选框样式统一
- 上传按钮样式统一
- 展开态和收起态继续向同一套组件系统收拢

### 建议满足

- 用户第一眼完全先看到图片
- 图片区顶部真正“清空”
- 底部控件虽然存在，但不显乱

## 10. 风险与权衡

### 风险 1

全部控件沉到底部后，底部可能显得拥挤。

应对：

- 控件尺寸统一但轻量
- 控件分左右组
- 只保留必要操作

### 风险 2

如果拖拽和勾选也沉到底部，局部交互密度会上升。

应对：

- 优先做统一分组
- 不追求每个按钮都显眼，追求整体秩序
