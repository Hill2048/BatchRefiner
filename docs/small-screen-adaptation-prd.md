# 小尺寸屏幕适配 PRD

## 1. 背景

当前 BatchRefiner 主界面以桌面端双栏布局为主，左侧配置栏默认宽度 `320px`，右侧任务工作区采用整屏铺开。实际使用中，在较小尺寸屏幕或较窄窗口下，出现以下问题：

- 左侧配置栏占比过高，右侧任务区被明显压缩。
- 页面整体使用整屏高度并限制外层滚动，导致部分区域只能局部滚动，用户容易误判为“内容被遮挡”。
- 任务卡片展开态、底部状态条、顶部区域在窄宽度下进一步挤压主内容。
- 用户在 13 寸笔记本、分屏窗口、PWA 小窗口场景下的可用性明显下降。

本次先定义一版小尺寸屏幕适配方案，确认后进入实现。

## 2. 问题定义

### 2.1 用户可见问题

- 小屏时左侧栏内容过长，底部操作区容易掉出第一屏。
- 主工作区宽度不足时，任务区标题、视图切换、卡片网格会被压缩得过紧。
- 展开任务卡片后，卡片内部“参考图 + 参数 + 文本编辑”区域横向拥挤，阅读和操作困难。
- 底部悬浮状态条始终居中悬浮，在小屏下会遮挡主内容底部。

### 2.2 根因判断

基于当前代码初步确认：

- `src/App.tsx` 采用固定桌面双栏结构，左栏初始宽度固定为 `320px`，拖拽范围为 `250px - 600px`。
- 页面主框架使用 `h-screen`、`overflow-hidden`，外层缺少针对窄屏的结构切换。
- `src/components/layout/Sidebar.tsx` 中存在较多固定块级区域和底部固定操作区，天然更适合宽侧栏而非窄侧栏。
- `src/components/workspace/TaskCard.tsx` 的展开态存在多个 `min-width` 约束，小屏下容易横向拥挤。
- `src/App.tsx` 的底部悬浮状态条在所有尺寸下都固定悬浮，未针对窄屏降级。

## 3. 产品目标

### 3.1 目标

- 让小尺寸屏幕下的核心操作可完整访问，不出现主要 UI 被遮挡或无法触达的情况。
- 保留当前桌面端设计风格，不做大范围视觉重构。
- 保证“导入任务 -> 设置全局参数 -> 查看任务 -> 执行任务”主流程在小屏下可顺畅完成。

### 3.2 非目标

- 本次不重做整套信息架构。
- 本次不新增复杂的移动端手势交互。
- 本次不处理超小手机竖屏的完整原生移动端适配。
- 本次不改动业务流程，仅调整布局、可视层级和交互呈现。

## 4. 适配范围

本次按窗口宽度分三档处理：

- 大屏桌面：`>= 1280px`
  保持现有双栏桌面布局为主，仅做少量弹性修正。
- 中小屏桌面：`1024px - 1279px`
  仍保留双栏，但左栏需收窄，部分组件压缩。
- 小屏/窄窗口：`<= 1023px`
  从固定双栏切换为“顶部主操作 + 抽屉/折叠侧栏 + 单列工作区”模式。

备注：这里的“小尺寸屏幕”优先指 13 寸笔记本、Windows 分屏、PWA 小窗口，不以手机浏览器为第一目标。

## 5. 目标用户场景

- 用户在 13 寸笔记本全屏使用应用。
- 用户在 Windows 桌面左右分屏，仅保留半屏宽度。
- 用户将 PWA 安装为桌面应用后，以较小窗口常驻使用。

## 6. 方案概述

### 6.1 布局策略

- `>= 1280px`
  维持当前桌面双栏。
- `1024px - 1279px`
  左侧栏默认宽度从 `320px` 收敛到更紧凑的宽度区间，右侧工作区优先保宽。
- `<= 1023px`
  左侧栏不再常驻，占位从主布局中移除，改为：
  - 顶部提供“打开配置面板”入口；
  - 配置内容以抽屉、覆盖层或折叠面板形式出现；
  - 主区域改为单列，自上而下展示任务区。

### 6.2 关键界面调整

#### A. 顶部栏

- 小屏时压缩左右留白。
- 项目名过长时允许截断，不挤压其他控件。
- “安装应用”按钮在小屏下允许仅图标或次级展示。

#### B. 左侧配置栏

- 小屏下从常驻栏改为抽屉式面板。
- 抽屉内部保留纵向滚动，确保底部执行区可触达。
- 模块顺序保持不变，减少学习成本。
- 底部操作区在抽屉内需始终可见或容易滚到。

#### C. 主工作区

- 小屏时顶部标题区改为可换行或纵向排列。
- “网格/列表”切换按钮不能挤压标题核心信息。
- 空状态提示区在小屏下降低内边距和高度，避免占满可视区。

#### D. 任务卡片

- 网格视图最小卡片宽度下调，避免窄屏只剩极少内容宽度。
- 展开态卡片在小屏下不再使用过宽的横向参数区，改为分段纵向堆叠。
- 全局参考图、任务参考图、专属参数这三块在小屏展开态下改为上下排列。
- 文本编辑区保持可读，不出现输入框过窄或按钮换行错乱。

#### E. 底部悬浮状态条

- 小屏下改为更紧凑形态。
- 优先考虑：
  - 折叠为更短的状态胶囊；
  - 或固定到右下角；
  - 或在小屏下并入顶部/侧栏，避免遮挡任务列表底部。

## 7. 详细需求

### 7.1 布局切换要求

- 窗口宽度进入小屏阈值后，布局应自动切换，无需刷新页面。
- 从小屏恢复到中大屏后，桌面布局应恢复正常。
- 切换过程中不丢失当前项目数据和输入内容。

### 7.2 侧栏交互要求

- 小屏必须存在明显的“打开配置”入口。
- 打开配置面板后，遮罩和关闭方式应清晰。
- 用户在配置面板内修改内容后，关闭面板不应丢失输入。

### 7.3 内容可见性要求

- 首屏必须能看到主工作区标题和当前任务区域。
- 任何主要操作按钮不能落到不可见区域且无明显滚动提示。
- 不允许出现横向整体溢出导致页面被裁切。

### 7.4 滚动要求

- 小屏下优先保证页面滚动逻辑简单。
- 避免出现外层不能滚、内层又不明显可滚的情况。
- 抽屉、主工作区、任务展开态的滚动行为要明确，不互相抢滚动焦点。

### 7.5 兼容性要求

- 适配 Edge 浏览器。
- 兼容 PWA 安装后的桌面窗口使用。
- 兼容 Windows 系统缩放场景下的常见窄视口。

## 8. 验收标准

### 8.1 必须满足

- 在 `1024px` 宽度及以下，左侧栏不再持续挤占主工作区宽度。
- 在 `1024px` 宽度及以下，用户可通过明确入口打开完整配置面板。
- 在 `1024px` 宽度及以下，首页首屏能看到当前任务区域，且不会被左栏或底部状态条主要遮挡。
- 在展开任务卡片时，核心编辑区不出现严重横向挤压或内容不可读。
- 页面不存在明显的横向滚动条。

### 8.2 建议满足

- 在 `1280px` 以下，整体观感仍然保持当前产品的轻量和精致风格。
- 抽屉/折叠态切换有轻量过渡动画，但不影响性能。
- 小屏下至少能在不频繁缩放窗口的情况下完成一次完整任务操作。

## 9. 设计原则

- 优先解决“能不能用”，再解决“像不像桌面大屏版本”。
- 优先保证主流程可达，不强求所有装饰元素都保留。
- 适配方案尽量复用现有组件，避免引入第二套完全独立页面。

## 10. 技术实现建议

以下为实现建议，不作为最终代码方案强绑定：

- 在 `App` 层引入响应式断点状态，控制主布局切换。
- 小屏下隐藏常驻 sidebar 容器，改为抽屉式 `Sidebar` 挂载。
- 调整 `TaskList` 顶部标题区和空状态区的响应式样式。
- 调整 `TaskCard` 展开态的参数区布局，减少固定 `min-width` 对小屏的压迫。
- 重构底部状态条在小屏下的展示策略。
- 尽量以 CSS 响应式类为主，少量配合 React 状态控制结构切换。

## 11. 风险与注意项

- 当前 sidebar 支持拖拽改宽，小屏切换为抽屉后，该能力需要自动禁用或仅保留在桌面模式。
- 任务卡片展开态信息较多，小屏下如果完全保留单屏展示，会继续拥挤，需要接受“纵向更长”的设计取舍。
- 如果继续保留底部悬浮状态条原样式，小屏遮挡问题难以彻底解决。

## 12. 建议实施顺序

1. 先完成主布局断点切换和小屏 sidebar 抽屉化。
2. 再完成任务区顶部和空状态压缩。
3. 然后处理 TaskCard 展开态的小屏纵向重排。
4. 最后处理底部状态条的小屏降级展示。

## 13. 待你确认的决策点

请优先确认下面 3 点，确认后即可进入实现：

1. 小屏模式是否接受“左侧栏改为抽屉面板，不再常驻显示”。
2. 底部悬浮状态条是否接受“小屏下缩短/折叠/移位”，而不是强行保留原样。
3. 任务卡片展开态是否接受“小屏下改为纵向堆叠”，而不是继续保持横向三列信息区。
