# 结果图区合并优先 PRD

## 1. 背景

当前展开态结果图区虽然已经做过多轮简化，但图片外侧仍然有较多独立层级：

- 外层结果容器
- 顶部原图 / 结果图切换条
- 独立主图片区
- 主图外的缩略图区
- 图片外侧的尺寸与状态信息

从用户感知上看，这一块仍然像“图片查看器 + 一圈外围 UI”，而不是“一张图本身承载全部必要信息”。

用户这次的目标很明确：

- 图片周围的一圈信息继续减少
- 能收进图片里的都收进图片里
- 包括缩略图也尽量并入图片区域
- 最终更接近“图片本身就是界面”的效果
- 这次还要进一步收，不只是合并，而是继续减少图片外的任何独立存在感
- 新原则是：能合并的就合并

## 2. 目标

- 把展开态结果图区改成“以图片为唯一主容器”的结构
- 将原图 / 结果图切换、尺寸标签、批次缩略图尽量内嵌到图片区域
- 消除图片外部多余的一圈卡片和分区感
- 保留必要功能，不牺牲可用性
- 把图外信息压缩到最低，只允许极少数必要内容留在图外
- 把原本分散的图片区、切换区、缩略图区合并成一个整体图片单元

## 3. 非目标

- 本次不改批次生成逻辑
- 本次不改 Lightbox 的核心交互逻辑
- 本次不改任务卡片下方参数区、提示词区、参考图区
- 本次不改列表态小卡片布局

## 4. 问题定义

当前结果图区的核心问题不是功能不够，而是信息分布位置不对。

### 4.1 层级过多

现在用户看到的是：

1. 一个结果区容器
2. 容器内部再放顶部切换条
3. 再放主图卡片
4. 再放底部缩略图区

结果是：

- 图片被外围 UI 包住
- 主图不够“直接”
- 用户第一眼还是先看到结构，再看到图

### 4.2 信息都在图外

当前很多与图片强相关的信息仍在图片外：

- 原图 / 结果图切换
- 批次缩略图
- 部分状态信息

这会让整个查看体验像“操作一个组件”，而不是“看一张图”。

### 4.3 缩略图和主图脱节

虽然缩略图已经缩小，但仍然在主图下方作为独立区域存在。

这导致：

- 主图和批次缩略图仍是两块
- 图片整体重心被切开
- 视觉上还是“上面一块，下面一块”

## 5. 设计原则

这版只遵守 4 条原则：

### 5.1 图片优先

所有必要信息依附图片，而不是图片依附组件。

### 5.2 外围归零

能取消的外部容器就取消，能并入图片的就并入图片。

### 5.3 控件弱化

控件仍保留，但默认像图片的附属浮层，而不是一排独立 UI。

### 5.4 单体感

最后看起来应该像“一个完整图片单元”，而不是“多块组件拼起来”。

### 5.5 图外清空

默认假设所有内容都应进图内。

只有在以下情况才允许留在图外：

- 会严重遮挡图片主体
- 会影响点击操作
- 属于任务卡片级信息而不是图片级信息

### 5.6 合并优先

如果两个区域服务的是同一件事，就不允许分开存在。

具体规则：

- 属于“看图”的信息必须合并进同一图片单元
- 属于“切图”的信息必须与图片本身合并
- 属于“图片结果说明”的信息不能再独立占一块
- 不允许为了排版舒服就继续拆成多块

## 6. 方案总览

目标结构调整为：

1. 一张主图作为唯一核心容器
2. 图片内部顶部合并切换与查看入口
3. 图片内部底部合并尺寸标签与批次缩略图
4. 图片外不再单独保留缩略图区、图片区卡片、信息条
5. 图片外只保留任务本身的信息，不再保留图片相关的信息层

也就是说：

- 外围信息向内收
- 图片内部承载操作
- 图片外只保留必要留白
- 只要属于“看图时需要的信息”，都默认放进图内
- 原本拆开的图片区功能统一合并成一个图片系统

## 7. 详细方案

## 7.1 主图容器

主图应成为唯一视觉核心。

### 调整方向

- 取消主图外再包一圈明显的“图片区卡片”
- 减少图片外部边框、留白、内衬
- 展开态结果区看起来更像一张带少量浮层信息的图片，而不是一个嵌套组件
- 能去掉的二层容器就去掉，不再保留“图外一圈面板”

### 目标状态

- 用户第一眼看到的是图
- 图片是整块结果区的绝对中心
- 外层只保留必要的极轻边界
- 图外不再出现明显的图片区盒子感
- 不再存在“一个图片区 + 一个图外附件区”的结构

## 7.2 顶部信息并入图片

目前顶部切换条在图片外，后续改为收进图片顶部。

### 收进内容

- 原图 / 结果图切换
- 放大查看按钮
- 如有必要，可保留已返回计数

### 展示方式

- 作为图片顶部浮层存在
- 透明度低，弱背景
- 不形成一整条厚工具栏
- 更像贴边浮层，而不是工具条

### 要求

- 默认不抢视觉焦点
- 仍然足够可点
- 不遮挡主体内容
- 控件数量必须继续减少，能不显示的就不显示
- 与图片形成同一层，而不是另起一块

## 7.3 尺寸标签保留并收进图片

尺寸标签继续保留，而且必须在图片内显示。

### 规则

- 仅结果图显示尺寸标签
- 原图不显示尺寸标签
- 放在图片右下角
- 使用轻浮层样式

### 目标

- 保留必要结果信息
- 不额外制造图片外层级

## 7.4 缩略图并入图片

这是本次最关键的一条。

当前缩略图还在主图下方单独占一行；这一版改成尽量并入图片内部。

### 方向

- 缩略图贴入图片底部区域
- 作为图片内底部胶片条存在
- 不再做成独立图片区外的缩略图区
- 与图片底边形成一体结构，不再像下面再挂一块
- 与尺寸标签、图片底边一起组成同一个底部信息层

### 样式建议

- 放在图片底部左侧或底部居中
- 继续保持小尺寸
- 使用半透明底衬或弱浮层，不单独起大容器
- 当前选中态保留高亮边框
- 比现在再小一档也可以接受，只要还能点

### 目标效果

- 用户理解为“这张图有多张批次结果”
- 而不是“这张图下面还有另一个模块”
- 看起来像贴在图片上的胶片导航，而不是一排按钮

## 7.4.1 缩略图与主图的合并规则

这次不再把缩略图视为“图片下方区域”，而是视为“图片的一部分”。

必须满足：

- 缩略图视觉归属主图
- 缩略图不再单独占据图外垂直空间
- 缩略图背景、边界、留白不能形成新的卡片感

## 7.5 主图与缩略图关系

收进图内后，主图与缩略图要形成一体结构。

### 结构目标

- 主图是主体
- 缩略图像底部胶片条
- 尺寸标签和缩略图不互相冲突

### 布局建议

可优先考虑：

- 右下角：尺寸标签
- 左下角：批次缩略图
- 顶部：原图 / 结果图切换 + 查看按钮

这样可以形成比较稳定的信息分布。

进一步内收建议：

- 顶部控件贴边
- 底部缩略图贴边
- 尺寸标签压到最角落
- 主图中间区域尽量不出现任何长期浮层
- 底部所有与结果相关的信息尽量并成一个层

## 7.6 状态信息收纳

对“已返回 x/y”这类信息，需要继续压缩。

### 方案建议

- 如果有多批次结果，显示在图片顶部轻浮层中
- 不再单独占用图片外的一段文案区域
- 单张结果不显示额外返回计数
- 如果顶部信息仍显多，优先把返回计数收成更短形式，例如 `2/4`

进一步规则：

- 返回计数不能单独成块
- 必须并入顶部或底部已有信息层

## 7.7 图外保留内容重新定义

这次进一步收的关键是重新定义什么可以留在图外。

### 图外允许保留

- 任务编号
- 任务标题
- 状态标签
- 任务级参数、提示词、参考图等非图片内容

### 图外不再允许保留

- 图片切换条
- 图片缩略图区
- 图片尺寸信息
- 图片结果计数条
- 明显的二层图片区包装
- 与图片强相关的独立工具区

## 7.8 必须合并的区域清单

这次实现时，以下区域默认必须合并：

### 必须合并 1

原图 / 结果图切换 + 查看按钮

合并目标：

- 变成图内顶部同一信息层

### 必须合并 2

主图 + 缩略图

合并目标：

- 变成同一个图片单元

### 必须合并 3

尺寸标签 + 结果图底部信息

合并目标：

- 变成图内底部同一信息层

### 必须合并 4

返回计数 + 顶部图片操作信息

合并目标：

- 不再各自存在，统一放入图内顶部轻信息层

## 7.9 禁止继续拆分

为了避免改着改着又变回多层结构，这次明确禁止：

- 禁止给缩略图再单独包一层图片区
- 禁止给尺寸标签再单独包一层信息盒
- 禁止给切换按钮再单独包一层工具条
- 禁止在图片外再保留“为了整齐”的独立图片区块

## 8. 交互要求

## 8.1 原图 / 结果图切换

- 保留
- 改为图内顶部浮层切换
- 不再单独占据图片外一行

## 8.2 缩略图切换

- 保留
- 点击缩略图切换当前主图
- 当前缩略图高亮仍保留
- 缩略图移动到图内底部

## 8.3 放大查看

- 保留
- 放在图内右上角轻按钮
- 不再占据图外空间

## 8.4 尺寸信息

- 保留
- 图内右下角
- 跟随当前结果图变化

## 9. 视觉层级重构

当前层级：

1. 外层容器
2. 顶部切换条
3. 主图卡片
4. 缩略图区

目标层级：

1. 极轻外层结果容器
2. 一张主图
3. 图内顶部合并信息层
4. 图内底部合并信息层

重点是从“图外分块”改成“图内浮层”。

进一步收后的目标是：

- 视觉上只能明显感知到“一张图”
- 其他信息都只是贴在图边缘的辅助层

## 10. 验收标准

### 必须满足

- 图片外部层级明显减少
- 顶部切换不再独立悬在图片外
- 缩略图不再作为独立图片区外模块存在
- 尺寸标签保留在图内右下角
- 整个结果区看起来更像一张带信息的图片，而不是多个小区域拼接
- 图外不再看到明显的“图片模块外框 + 缩略图区 + 切换条”三段结构
- 凡是属于图片结果的信息，默认都能在同一个图片单元内找到

### 建议满足

- 用户第一眼先看到图，而不是周围 UI
- 主图、缩略图、尺寸、切换形成单体感
- 整体高度更紧凑
- 视觉上比当前版本至少再少一层外框感

## 11. 风险与权衡

### 风险 1

信息都收进图内后，可能遮挡图片主体。

应对：

- 所有浮层都做轻量透明背景
- 缩略图尺寸继续小型化
- 控件只占边角

### 风险 2

缩略图收进图内后，如果摆放不当，会显得拥挤。

应对：

- 优先做单排小缩略图
- 只保留必要数量
- 间距尽量小但可点击

### 风险 3

如果图内信息太多，可能从“简化”变成“全部叠到图上”。

应对：

- 只保留 4 类核心信息：
- 原图 / 结果图切换
- 放大按钮
- 尺寸标签
- 批次缩略图

## 12. 待确认点

请确认下面 4 点：

1. 原图 / 结果图切换要收进图片顶部浮层，不再单独放在图片外，对吗。
2. 批次缩略图要收进图片底部区域，不再单独占图片外一行，对吗。
3. 尺寸标签继续保留在图片右下角，且只在结果图显示，对吗。
4. 如果图内信息太多，优先保留缩略图和尺寸，其他信息继续弱化，对吗。

## 12.1 本次新增确认原则

再补 2 个总原则，请一起确认：

1. 只要两个区域都服务“看当前结果图”，就优先合并，不再保留成两块。
2. 如果合并后仍可点击、可读，就不允许因为“整齐”继续拆开。

## 13. 这次进一步收的具体执行顺序

1. 先去掉图片外单独的切换条容器
2. 再去掉图片外单独的缩略图区容器
3. 把尺寸标签、缩略图、切换按钮全部压进图片边缘
4. 最后再删掉图片外剩余不必要的边框和留白
5. 最后检查是否还有本可以合并却仍拆开的块，继续合并
